# CODEGEN_SPEC.md — Code Generation Engine Specification

> `CodeGenerator.ts` must handle every case in this document. Each section includes the input `JoinState` and the exact expected output string. Use these as your Jest test cases.

---

## Core Rules

1. `CodeGenerator.ts` has **zero VS Code API imports** — it is pure TypeScript.
2. It takes a `JoinState` and returns a `string`.
3. All variable names come from `JoinState` — never hardcoded.
4. Generated code must be valid, runnable Python on the first try.
5. Add a single-line header comment: `# Generated by Visual Join Builder`

## Case 1: Single Left Join — Pandas

**Input JoinState:**
```json
{
  "dialect": "pandas",
  "outputName": "result_df",
  "tables": [
    { "name": "df_users", "columns": [
      { "name": "id", "dtype": "int64", "nullable": false },
      { "name": "name", "dtype": "object", "nullable": false }
    ]},
    { "name": "df_orders", "columns": [
      { "name": "order_id", "dtype": "int64", "nullable": false },
      { "name": "user_id", "dtype": "int64", "nullable": false },
      { "name": "total", "dtype": "float64", "nullable": false }
    ]}
  ],
  "joins": [{
    "id": "e1",
    "leftTable": "df_users",
    "leftColumn": "id",
    "rightTable": "df_orders",
    "rightColumn": "user_id",
    "joinType": "left"
  }],
  "selectedColumns": [
    { "table": "df_users", "column": "id", "alias": null },
    { "table": "df_users", "column": "name", "alias": null },
    { "table": "df_orders", "column": "total", "alias": "order_total" }
  ]
}
```

**Expected Output:**
```python
# Generated by Visual Join Builder
result_df = df_users.merge(
    df_orders,
    how='left',
    left_on='id',
    right_on='user_id'
)
result_df = result_df[['id', 'name', 'total']]
result_df = result_df.rename(columns={'total': 'order_total'})
```

## Case 2: No Column Selection (Select All) — Pandas

When `selectedColumns` is empty, omit the column filter step entirely.

**Expected Output:**
```python
# Generated by Visual Join Builder
result_df = df_users.merge(
    df_orders,
    how='inner',
    left_on='id',
    right_on='user_id'
)
```

## Case 3: Column Name Collision — Pandas

When both tables have a column with the same name (e.g., `created_at`), the engine MUST use explicit `suffixes` to prevent Pandas from appending `_x`/`_y` silently.

**Collision detection rule:** Scan `tables[0].columns` and `tables[1].columns`. If any non-join-key column name appears in both, a collision exists.

**Expected Output:**
```python
# Generated by Visual Join Builder
# NOTE: Column name collision detected: created_at
result_df = df_users.merge(
    df_orders,
    how='left',
    left_on='id',
    right_on='user_id',
    suffixes=('_users', '_orders')
)
```

## Case 4: Chained Joins (A → B → C) — Pandas

Multi-join resolution must be topologically ordered. The engine builds a dependency graph from `JoinState.joins` and resolves the merge sequence.

**Input joins:**
```json
[
  { "leftTable": "df_a", "leftColumn": "id", "rightTable": "df_b", "rightColumn": "a_id", "joinType": "inner" },
  { "leftTable": "df_b", "leftColumn": "id", "rightTable": "df_c", "rightColumn": "b_id", "joinType": "left" }
]
```

**Expected Output:**
```python
# Generated by Visual Join Builder
_step1 = df_a.merge(
    df_b,
    how='inner',
    left_on='id',
    right_on='a_id'
)
result_df = _step1.merge(
    df_c,
    how='left',
    left_on='id',
    right_on='b_id'
)
```

Note: Intermediate variable named `_step{n}` (underscore prefix signals it's internal). Final result uses `outputName`.

## Case 5: Cross Join — Pandas

Cross joins have no ON condition. Generate a cartesian product call.

**Expected Output:**
```python
# Generated by Visual Join Builder
result_df = df_a.merge(df_b, how='cross')
```

## Case 6: Type Mismatch Warning — Pandas

If `leftColumn.dtype !== rightColumn.dtype`, emit a warning comment and a cast line.

**Detection rule:** Compare `dtype` strings from `ColumnSchema`. If they differ and one is `'object'` and the other is a numeric type, generate a cast.

**Expected Output:**
```python
# Generated by Visual Join Builder
# WARNING: Type mismatch on join key — df_orders.user_id is int64, df_users.id is object
# Casting df_orders.user_id to object to match
df_orders = df_orders.copy()
df_orders['user_id'] = df_orders['user_id'].astype(object)
result_df = df_users.merge(
    df_orders,
    how='left',
    left_on='id',
    right_on='user_id'
)
```

## Case 7: DuckDB Dialect

**Expected Output for the same left join as Case 1:**
```python
# Generated by Visual Join Builder
import duckdb

query = """
SELECT
    df_users.id,
    df_users.name,
    df_orders.total AS order_total
FROM df_users
LEFT JOIN df_orders ON df_users.id = df_orders.user_id
"""
result_df = duckdb.sql(query).df()
```

**DuckDB rules:**
- Always include `import duckdb` at the top.
- Use `duckdb.sql(query).df()` — the `.sql()` method is the documented standard. (`.query()` is an alias and also works, but use `.sql()`.)
- DuckDB automatically scans Python local variables by name — `df_users` in SQL refers directly to the Pandas DataFrame variable in scope. No registration needed.
- Use `.df()` to convert the result to a Pandas DataFrame — not `.to_df()` (that is the old API).
- Table aliases: use first letter of table name (e.g., `df_users` → `u`, `df_orders` → `o`). If ambiguous, use first two letters.

## Case 8: PySpark Dialect

**Expected Output for the same left join as Case 1:**
```python
# Generated by Visual Join Builder
result_df = df_users.join(
    df_orders,
    on=df_users['id'] == df_orders['user_id'],
    how='left'
).select(
    df_users['id'],
    df_users['name'],
    df_orders['total'].alias('order_total')
)
```

**PySpark rules:**
- Use `.join()` chaining, never `pd.merge()`.
- Always use the `.select()` clause even when all columns are selected (use `'*'` then).
- The `how` parameter maps: `outer` → `'outer'`, `inner` → `'inner'`, etc.
- For chained joins, chain `.join()` calls on the same expression.

---

## Case 9: All Columns Selected Explicitly (All Checkboxes Checked)

When all columns from all tables are selected (every column in every table appears in `selectedColumns`), treat it the same as "no selection" — omit the column filter step. This avoids generating needless boilerplate.

**Detection rule:** `selectedColumns.length === sum of all column counts across all tables`

## Edge Cases Summary Table

| Scenario | Pandas behavior | DuckDB behavior | PySpark behavior |
|---|---|---|---|
| No joins defined | Return empty string | Return empty string | Return empty string |
| Single table, no joins | Return empty string | Return empty string | Return empty string |
| Cross join | `how='cross'`, no `on` | `CROSS JOIN`, no `ON` | `how='cross'`, `on=None` |
| All columns selected | No filter step | `SELECT *` | `.select('*')` |
| Column name collision | `suffixes=` param | Table alias prefix | Table ref prefix |
| Type mismatch | Cast + warning comment | `CAST()` in SQL | `.cast()` in select |
| Chained (3+ tables) | `_step{n}` intermediates | Single SQL query | Chained `.join()` |
| Alias on column | `.rename(columns=...)` | `col AS alias` | `.alias('name')` |
| Null output name | Default to `result_df` | Default to `result_df` | Default to `result_df` |
